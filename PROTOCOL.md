# MotionLink — протокол клиент → сервер

Этот документ описывает проводной протокол, который реализует клиент MotionLink (iOS-приложение). Серверу нужно совместимо ответить на discovery и принимать поток данных датчиков.

Транспорт **только UDP**. Никакого TCP/WebSocket/HTTP.

---

## 1. Discovery (поиск сервера в локальной сети)

### Что делает клиент

После запуска приложение каждую **1 секунду** шлёт UDP-датаграмму:

| Поле | Значение |
|---|---|
| Destination | `255.255.255.255:58930` (limited broadcast) |
| Source | произвольный эфемерный UDP-порт, выбранный ОС (`bind` на `0.0.0.0:0`) |
| Socket options | `SO_BROADCAST=1`, `SO_REUSEADDR=1` |
| Payload | ASCII-строка `MOTIONLINK_DISCOVER\n` (20 байт: 19 символов + `0x0A`) |

Тот же сокет используется для приёма ответа — слушать новый порт не нужно, ответ должен прийти на адрес источника пробы.

### Что должен делать сервер

1. Биндиться на `0.0.0.0:58930` UDP (или конкретный интерфейс), `SO_REUSEADDR=1`.
2. Принимать датаграммы. Если payload начинается с `MOTIONLINK_DISCOVER` (можно сравнивать prefix, без учёта завершающего `\n`) — отправить ответ **на адрес источника** (`recvfrom`-овский `sockaddr`), то есть на эфемерный порт клиента.
3. Payload ответа — однострочный JSON-объект:

```json
{"name": "имя сервера", "port": 58930}
```

Оба поля **опциональные**:

| Поле | Тип | Назначение |
|---|---|---|
| `name` | string | Отображается в UI карточкой сервера. Если отсутствует, клиент покажет «Сервер». |
| `port` | integer (UInt16) | UDP-порт, на который клиент будет слать данные датчиков **после** того, как пользователь нажал «Подключиться». Если отсутствует — клиент использует `58930`. |

Минимальный валидный ответ: `{}`. В этом случае клиент будет считать, что сервер слушает данные на том же `58930`.

### Поведение клиента

- IP сервера клиент **не достаёт из JSON** — он берёт его из `sockaddr` пакета ответа. Серверу не нужно сообщать свой IP в payload.
- Любой UDP-пакет на эфемерном порту, который начинается с `MOTIONLINK_DISCOVER`, **игнорируется** (защита от рефлексии broadcast'а самим клиентом).
- Если ответ пришёл от того же IP, который уже в списке, — он не дублируется.
- Discovery работает бесконечно, пока пользователь не выберет сервер. Таймаута нет.

### Пример обмена

```
client → 255.255.255.255:58930   "MOTIONLINK_DISCOVER\n"
server → 192.168.1.42:53217      {"name":"dev-laptop","port":58930}
```

После этого пользователь видит карточку `dev-laptop` с `192.168.1.42:58930` и кнопку «Подключиться».

## 2. Поток данных датчиков (после подтверждения пользователя)

### Транспорт

- Клиент открывает UDP-соединение на `host:port`, где `host` — IP из discovery-ответа, `port` — поле `port` из JSON-ответа (или `58930`, если не указан).
- Сокет UDP, без какого-либо рукопожатия. Сервер просто должен слушать этот порт.
- Частота отправки: **~30 Гц** (один пакет каждые ~33 мс).
- Размер пакета: ~200–300 байт (один JSON-объект на датаграмму).
- Без подтверждений: клиент **не ждёт ответа**, server-side losses допустимы (UDP best-effort).
- Без keep-alive: если приложение в фоне или экран закрыт — поток прекращается без уведомления.
- Если пользователь нажмёт «Отключиться» — клиент закрывает соединение и возвращается к discovery.

### Формат пакета

Один JSON-объект на датаграмму. Компактный (без переносов и лишних пробелов), UTF-8.

```json
{
  "t": 1779993600.123456,
  "roll": -12.4,
  "pitch": 34.1,
  "yaw": 87.2,
  "acceleration": {"x": 0.012, "y": -0.143, "z": 0.981},
  "gravity":      {"x": 0.000, "y": 0.000, "z": -1.000},
  "rotationRate": {"x": 0.002, "y": -0.001, "z": 0.004}
}
```

| Поле | Тип | Единица | Описание |
|---|---|---|---|
| `t` | double | секунды с Unix epoch (POSIX time), с дробной частью | `Date().timeIntervalSince1970` на момент чтения сенсоров |
| `roll` | double | **градусы** | Поворот вокруг продольной оси устройства, диапазон ≈ [-180; 180] |
| `pitch` | double | градусы | Поворот вокруг поперечной оси, диапазон ≈ [-90; 90] |
| `yaw` | double | градусы | Поворот вокруг вертикальной оси, диапазон ≈ [-180; 180] |
| `acceleration` | object `{x,y,z}` | **g** (1g ≈ 9.81 м/с²) | Линейное ускорение **без гравитации** (CoreMotion `userAcceleration`) |
| `gravity` | object `{x,y,z}` | g | Вектор гравитации в системе координат устройства (магнитуда ≈ 1) |
| `rotationRate` | object `{x,y,z}` | **рад/с** | Угловая скорость (CoreMotion `rotationRate`) |

Все числа — JSON numbers (double), могут быть отрицательными, могут содержать `e`-нотацию.

### Система координат устройства (CoreMotion device body frame)

Когда iPhone держат в **портретной ориентации, экраном к себе**:

- **X** — вправо (по горизонтали экрана)
- **Y** — вверх (по вертикали экрана, к динамику)
- **Z** — из экрана наружу, к лицу пользователя

Поэтому в покое экраном вверх (телефон лежит на столе) ожидаем `gravity ≈ {x:0, y:0, z:-1}`.

Поля `roll`/`pitch`/`yaw` — углы Эйлера ориентации этой device frame относительно опорной системы CoreMotion (`CMAttitudeReferenceFrame.xArbitraryZVertical`: Z вертикально вверх, X произвольно в горизонтальной плоскости и зафиксирован в момент первого замера). `yaw` поэтому абсолютным компасом **не является** — это смещение относительно стартовой ориентации.

### Поведение клиента при сбоях сети

- Если `sendto` возвращает ошибку — клиент логирует, но не падает и не переподключается; продолжает слать следующий пакет.
- Зелёный индикатор в HUD загорается, когда `NWConnection` переходит в `.ready`, и гаснет на `.failed`/`.cancelled`. Это не значит, что сервер реально слушает — UDP без рукопожатия.

## 3. Минимальный референс-сервер на Python

Для проверки клиента достаточно:

```python
import socket, json

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
sock.bind(("0.0.0.0", 58930))
print("listening on 58930")

while True:
    data, addr = sock.recvfrom(2048)
    if data.startswith(b"MOTIONLINK_DISCOVER"):
        sock.sendto(b'{"name":"dev","port":58930}', addr)
    else:
        try:
            packet = json.loads(data)
            print(f"t={packet['t']:.3f} "
                  f"r/p/y=({packet['roll']:+6.1f},{packet['pitch']:+6.1f},{packet['yaw']:+6.1f}) "
                  f"a={packet['acceleration']}")
        except json.JSONDecodeError:
            print("bad packet:", data[:60])
```

Этот сервер:
- слушает discovery и данные на одном сокете `58930`;
- отвечает на пробу клиента с именем `dev`;
- печатает входящие пакеты данных.

## 4. Сводка для серверной стороны

- **Слушать**: `0.0.0.0:58930/udp` (или другой порт, который сервер укажет в discovery-ответе в поле `port`).
- **Discovery-пробы**: пакеты с префиксом `MOTIONLINK_DISCOVER`, отвечать JSON-объектом на адрес источника.
- **Данные**: однострочные JSON-объекты с полями `t, roll, pitch, yaw, acceleration{x,y,z}, gravity{x,y,z}, rotationRate{x,y,z}`, ~30 пакетов/сек на клиента.
- **Никакого ответа** на пакеты данных клиенту слать не нужно — он их не ждёт и не читает.
- Один клиент = один поток; если нужно поддерживать несколько устройств — различать по `addr` из `recvfrom`.
